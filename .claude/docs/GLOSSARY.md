# Glossary

## Facts

| Term | Meaning | Source |
|------|---------|--------|
| arr / *arr | Shorthand for Sonarr (TV) and Radarr (movies) — one unified v3 client, `ArrClient`, built from two frozen spec instances (`SONARR_SPEC` / `RADARR_SPEC`) so a single class serves both APIs | `src/mediaman/services/arr/base.py`, `src/mediaman/services/arr/spec.py` |
| NZBGet | The Usenet download-queue backend, integrated over its native **JSON-RPC** API (`/jsonrpc` endpoint) — not XML-RPC despite the README's wording. `NzbgetClient` wraps `get_status`/`get_queue`/`is_reachable`, capped at 1 MiB responses | `src/mediaman/services/downloads/nzbget.py` (`NzbgetClient`) |
| media item | One row in `media_items`: a movie or a single TV season (never a whole show or an episode). `id` equals the Plex rating key of that movie/season and is the only immutable column — everything else (title, `show_title`, `season_number`, `plex_library_id`, `added_at`, …) is refreshed on every scan's upsert | `src/mediaman/db/schema_definition.py` (`media_items`), `src/mediaman/scanner/repository/media_items.py` (`upsert_media_item`) |
| Scan phases: evaluate / upsert / delete | The three pure/impure stages a library scan drives an item through, split into their own sub-package: `evaluate_item` (age + inactivity → `'skip'` \| `'schedule_deletion'`), `upsert_item` + `schedule_deletion` (DB write + HMAC keep-token mint), `remove_orphans` (fail-closed orphan pruning) | `src/mediaman/scanner/phases/evaluate.py`, `src/mediaman/scanner/phases/upsert.py`, `src/mediaman/scanner/phases/delete.py` |
| Scheduled action | A row in `scheduled_actions`, one per media item with an active deletion decision. `action` is one of the three `ACTION_*` constants below; `execute_at` is the deadline (`NULL` for forever); a partial unique index allows at most one active row per `media_item_id` | `src/mediaman/db/schema_definition.py` (`scheduled_actions`), `src/mediaman/core/scheduled_action_kinds.py` |
| `protected_forever` (protected-forever) | `ACTION_PROTECTED_FOREVER` — the media item is kept indefinitely; `execute_at` is `NULL` and it is never re-evaluated for deletion | `src/mediaman/core/scheduled_action_kinds.py` |
| `snoozed` | `ACTION_SNOOZED` — the media item is protected only until `execute_at` (one of the finite `VALID_KEEP_DURATIONS`: 7 / 30 / 90 days); deletion eligibility resumes after that instant | `src/mediaman/core/scheduled_action_kinds.py`, `src/mediaman/web/models/_common.py` (`VALID_KEEP_DURATIONS`) |
| `scheduled_deletion` | `ACTION_SCHEDULED_DELETION` — the default state written by the scanner: the item is due for deletion at `execute_at` unless a keep-token / keep API call cancels it first | `src/mediaman/core/scheduled_action_kinds.py` |
| kept / keep-token | "Kept" is the user-facing state for any item currently `protected_forever` or `snoozed` with a future `execute_at` (surfaced at `GET /api/kept`). A keep-token is the HMAC-signed link mailed in the newsletter / shown in the UI; `generate_keep_token`/`validate_keep_token` sign `{media_item_id, action_id, exp}`, and only the SHA-256 hash is ever stored (`scheduled_actions.token_hash`) — `schedule_deletion` nulls the raw `token` column after hashing. Redemption is single-use via `keep_tokens_used` | `src/mediaman/web/routes/kept.py`, `src/mediaman/crypto/tokens.py` (`generate_keep_token`, `validate_keep_token`), `src/mediaman/db/schema_definition.py` (`keep_tokens_used`) |
| Keep API (`POST /api/media/{id}/keep`) | Applies a keep decision (`resolve_keep_decision`): `duration="forever"` → `protected_forever`; any other valid duration → `snoozed` with `execute_at = now + days`. Guarded UPDATEs (`apply_keep_snooze` / `apply_keep_forever`) never commit themselves — the route owns the transaction | `src/mediaman/services/scheduled_actions/_types.py` (`resolve_keep_decision`), `src/mediaman/services/scheduled_actions/_mutations.py` |
| delete intent | A durability row in `delete_intents` (`media_item_id`, `target_kind` = `"radarr"`\|`"sonarr"`, `target_id`, `started_at`, `completed_at`, `last_error`) opened **before** the outbound Radarr/Sonarr delete call and closed after local DB cleanup, so a crash mid-delete is reconciled at the next boot instead of silently orphaning the *arr* side | `src/mediaman/web/repository/delete_intents.py` (`_record_delete_intent`, `reconcile_pending_delete_intents`), `src/mediaman/db/schema_definition.py` (`delete_intents`) |
| Two-phase on-disk delete | `delete_status` on `scheduled_actions` moves `'pending'` → `'deleting'` (committed **before** the `rm`) → the row is dropped entirely and an `audit_log` row with `action="deleted"` is written. `recover_stuck_deletions` reconciles any row still `'deleting'` at startup by checking whether the file survived | `src/mediaman/scanner/deletions.py` (`DeletionExecutor`, `recover_stuck_deletions`) |
| abandon | Unmonitoring a stuck Radarr/Sonarr search so it stops retrying: `abandon_movie` / `abandon_series` / `abandon_seasons` unmonitor via the *arr* client and clear the search-throttle row only on full success (token `0` = "the movie itself" for `abandon_seasons`) | `src/mediaman/services/downloads/abandon.py` |
| auto-abandon | The unattended policy version of abandon: fires when `auto_abandon_enabled` is set **and** an item has sat `searching` for 14 days (`_AUTO_ABANDON_AFTER_SECONDS`) **and** its release is at least 30 days old (`_AUTO_ABANDON_RELEASE_GRACE_SECONDS`). The manual "Abandon" button in the UI appears earlier, after 10 hours (`_ABANDON_BUTTON_VISIBLE_AFTER_SECONDS`). A `sec:auto_abandon.fired` audit row is written **before** the destructive unmonitor so a compromised-settings attack stays discoverable | `src/mediaman/services/arr/auto_abandon.py` |
| redownload | `POST /api/media/redownload` — re-adds a previously deleted item to Radarr/Sonarr and triggers a search, recording the action via `record_redownload`. The newsletter separately mints a per-recipient redownload token/URL (`_mint_rec_tokens`) only when a stable `tmdb_id` resolves | `src/mediaman/web/routes/library_api/redownload.py`, `src/mediaman/web/repository/library_api.py` (`record_redownload`), `src/mediaman/services/mail/newsletter/subscribers.py` (`_mint_rec_tokens`) |
| newsletter | The weekly per-subscriber digest email: scheduled-deletion items (with keep-token links), recently-deleted items (with redownload links), disk/reclaimed-space stats, and AI recommendation cards. Entrypoint `send_newsletter(conn, secret_key, dry_run=False, grace_days=14, *, recipients=None, mark_notified=True)`, dispatched only from the scanner pipeline (never the request layer) | `src/mediaman/services/mail/newsletter/__init__.py` (`send_newsletter`) |
| subscriber | A row in `subscribers` (`email`, `active`) — the newsletter recipient list. `try_add_subscriber` closes the concurrent-insert race with `BEGIN IMMEDIATE` + a unique-index fallback; unsubscribe is via a signed token carrying the email, never a query parameter, so PII never lands in server access logs | `src/mediaman/db/schema_definition.py` (`subscribers`), `src/mediaman/web/repository/subscribers.py`, `src/mediaman/services/mail/newsletter/subscribers.py` (`_load_subscribers`) |
| recommendation | A row in `suggestions`, produced by the OpenAI recommendations pipeline: prompt assembly from Plex watch-history/ratings → `call_openai` → `parse_recommendations` (prompt-injection validated) → TMDB/OMDb enrichment → one transactional DELETE + INSERT into `suggestions` per refresh batch (`batch_id` = fixed-width `YYYY-MM-DD`). Manual refresh is cooldown-gated at 24h (`RECOMMENDATION_REFRESH_COOLDOWN_HOURS`) | `src/mediaman/db/schema_definition.py` (`suggestions`), `src/mediaman/services/openai/recommendations/persist.py` (`refresh_recommendations`), `src/mediaman/services/openai/recommendations/throttle.py` |
| poster | A show/movie/season artwork URL. TMDB posters are shaped at `w300` for cards (`shape_card`) and upgraded to `w500` for outbound emails; the `/api/poster/{rating_key}` route is an SSRF-safe proxy (HTTPS/443-only, host-allowlisted, DNS-re-resolved on every request) with an on-disk LRU cache; `sign_poster_url` mints the HMAC-signed poster link used outside an authenticated session | `src/mediaman/services/media_meta/_tmdb_shapes.py` (`shape_card`, `_POSTER_BASE_W300`), `src/mediaman/web/routes/poster/` (`fetch.py`, `cache.py`), `src/mediaman/crypto/tokens.py` (`sign_poster_url`, `generate_poster_token`) |
